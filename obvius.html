Obvius Acquisuite

Python, Web2py, Heroku, Elastic Beanstalk, S3, DynamoDB, Postgres, Selenium

A web application to accept logfiles from field servers via HTTP push

Github Repo

The Setup

One day I heard the new hire had some programming experience. I wasn't working as a software developer, so I was excited to have a coworker that also liked to code. I enthusiastically greated him only to find out that he had only "worked with" programmers before. My bubble was burst, but it turns out he would end up giving me a fantastic excuse to shun the core responsibilties of my job and program instead :)

Apparently he had part ownership in his previous company called Crunch Energy, which was to be dissolved, and he was bringing their biggest client with him. My company at the time decided to pickup the service line, but rejected the opportunity to buy the existing software used to provide the service. I don't actually know why, but I'd like to think it was because it was written in PHP.

The Service

The service was referred to as Bill Auditing and Tenant Submetering (BATS) and consisted of combining monthly utility data and quarter-hourly submeter data to generate energy bills for the client's tenants. The utility data was being collected using a browser plugin called iMacros that was able to record and parameterize mouse movements and keyboard input in a browser session (which I replaced with python and selenium). And the submeter data was being collected from field servers by the old company's software which was going away soon.

Transition Period

The first thing we needed to do was figure out how to get data off of the field servers without going through the old Crunch Energy servers. We tried using a SaaS from the server manufacturer called Building Manager Online but it didn't fulfill all our needs. We tried using a desktop utility written in C, also provided by the manufacturer, but the port was hard coded and therefore didn't work for all of our installations (I actually did try to request the original files and update/compile/make the script myself with a paramaterized port, but it didn't pan out). Eventually I said, forget this, I'm sure I can make something myself.

Taking Matters Into My Own Hands

I used the technologies I was most comfortable with at the time and setup a web2py app in order to accept logfiles via http push from the field servers. I tried both Heroku and Elastic Beanstalk for hosting. The servers were able to be configured to send logfiles to four different places so I was able to point one at both instances of my application for testing without causing any disruptions. All of the documentation I found for the servers was specific to PHP, but after a great deal of trial and error I finally arrived at this line of code that let me treat the logfiles sent by the field servers as python file objects, which was a pretty exciting moment for me.

file_handle=gzip.GzipFile( fileobj=io.BytesIO( field_storage_object.value ) , mode='r' )

Step 2: Get Carried Away

After I finally had a familiar object to work with, I was easily able to start saving the logfiles using AWS S3. I also took care of some more administrative stuff like creating ancillary database tables and additional views for my coworkers to more easily inferface with the application. But I also wanted to show off to the guy that came from Crunch Energy so I started putting the data into DynamoDB as well and creating some live, publically accessible graphs, as if to say I can totally just recreate your entire previous company's software product myself. To keep karmic balance in equilibrium, I was punished for my hubris by racking up $30/mo to my personal AWS account before realizing and deactivating DynamoDB, which was never a necessary part of the project anyway.

And They All Lived Happily Ever After

I cleaned up the code and moved the project over to company accounts I created for Heroku, Github, and AWS. And with that, our reliance on Crunch Energy was no more!
